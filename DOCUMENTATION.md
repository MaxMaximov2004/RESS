## Обзор проекта

**Название**: BestLand
**Аббревиатура**: Real Estate Solution System (RESS)
**Тип**: Веб-приложение
**Цель**: Управление объявлениями о недвижимости
**Целевая аудитория**: Богатые люди

### Основные возможности:

- Регистрация и аутентификация пользователей (email/password + Google OAuth)
- Создание, редактирование и удаление объявлений о недвижимости
- Поиск и фильтрация объявлений
- Управление профилем пользователя
- Загрузка изображений объектов недвижимости
- Адаптивный дизайн для всех устройств

## Технологический стек

### Обоснование

Для реализации был выбран стек MERN (MongoDB, Express.js, React, Node.js) поскольку он обеспечивает оптимальное сочетание производительности, гибкости и скорости разработки при создании современных веб-приложений, работающих с динамическими данными недвижимости.

#### Единый стек для клиентской и серверной части

Все компоненты приложения — клиентская часть и серверная логика — используют единый язык программирования JavaScript. Это упрощает взаимодействие между frontend и backend, ускоряет процесс разработки и снижает сложность поддержки кода.

#### Масштабируемость и расширяемость архитектуры

Стек позволяет быстро собрать MVP (Minimum Viable Product). Обеспечивает лёгкость расширения системы: можно добавлять новые модули (фильтры, карты, избранное, загрузку медиа) и изменять архитектуру без значительных переработок. Это позволяет поддерживать рост функционала и адаптировать систему под новые требования

#### Развитая экосистема и поддержка сообщества

Большое количество библиотек, плагинов и готовых решений сокращает время разработки и повышает надёжность итоговой системы.

#### Гибкость хранения данных (MongoDB)

Объекты недвижимости могут иметь разные структуры данных. Документная модель MongoDB легко адаптируется под такие особенности и ускоряет расширение системы без сложных миграций схем.

#### Express.js как удобный backend-фреймворк

Express.js предоставляет минималистичную, но гибкую основу для создания REST API. Он подходит для реализации поиска, фильтрации, авторизации пользователей и управления объявлениями.

#### React как инструмент для динамичного интерфейса

React обеспечивает быстрый и отзывчивый UI, поддерживает обновление данных без перезагрузки страницы и позволяет строить масштабируемые интерфейсы: фильтры, карточки объектов, личный кабинет пользователя.

#### Производительность Node.js

Неблокирующая модель обработки запросов в Node.js обеспечивает высокую производительность при работе с большим количеством одновременных обращений, что актуально для каталогов недвижимости и поисковых систем.

#### Итого

Стек MERN является оптимальным выбором для информационной системы в сфере недвижимости благодаря единому языку разработки, гибкому хранению данных, высокой производительности и удобству создания динамичных пользовательских интерфейсов.

### Архитектурные паттерны

#### **MVC (Model-View-Controller)**

Для реализации был выбран архитектурный паттерн MVC (Model-View-Controller), который обеспечивает чёткое разделение ответственности между хранением данных, бизнес-логикой и отображением информации. Такой подход особенно важен для сложных веб-приложений с динамическими данными, например, для каталогов недвижимости, где требуется одновременно управлять большим объёмом информации и предоставлять удобный пользовательский интерфейс.

##### Model (`api/models/`)

Представляет собой слой работы с данными и бизнес-логикой. В нашем приложении это сущности и модели `MongoDB` (User, Listing) и их взаимодействие с базой данных через ODM (`Mongoose`) или прямые запросы.

##### Controller (`api/controllers/`)

Обрабатывает запросы от пользователя и управляет взаимодействием между `Model` и `View`. В нашем приложении это `REST API` на базе `Express.js` и `Node.js`, маршруты и функции, отвечающие за создание, обновление, удаление и получение объектов недвижимости

##### View (`client`)

Отвечает за отображение информации и взаимодействие с пользователем. В нашем приложении это клиентская часть на `React`, которая формирует динамический интерфейс: каталоги объектов, фильтры, добавления и редактирования данных, управление профилем пользователя

#### **Component-Based Architecture**

В основе frontend лежит компонентный подход, который позволяет строить интерфейс из независимых, переиспользуемых блоков. Такой подход обеспечивает гибкость, масштабируемость и простоту сопровождения приложения.

##### Переиспользуемые UI-компоненты

Компоненты разрабатываются как независимые и самодостаточные элементы интерфейса. Они инкапсулируют свою разметку, стили и локальную логику, что позволяет использовать их повторно на разных страницах и в разных частях приложения.
Примеры: кнопки, формы, карточки товаров, модальные окна.

_Преимущества_: уменьшение дублирования кода, единый стиль и поведение интерфейса, лёгкость тестирования.

##### Страницы как композиция компонентов

Каждая страница приложения строится из комбинации базовых и сложных компонентов, формируя целостный интерфейс. Такой подход позволяет быстро собирать новые страницы и изменять существующие, просто переставляя или заменяя компоненты.

_Преимущества_: модульность, лёгкость внесения изменений, упрощённое масштабирование.

##### Разделение логики и представления

Логика управления состоянием и взаимодействия с `API` отделена от визуальной части. Это достигается использованием React-хуков, контейнерных компонентов или специальных сервисов/контекстов для работы с данными.

_Преимущества_: интерфейс остаётся чистым и декларативным, бизнес-логика централизована, код становится более предсказуемым и поддерживаемым.

#### **State Management with `Redux`**

Для управления глобальным состоянием приложения на клиентской части используется `Redux Toolkit`, который упрощает работу с `Redux` и снижает шаблонный код. Этот подход обеспечивает предсказуемость состояния, масштабируемость и централизованное управление данными.

##### Централизованное хранилище (store)

Все данные, которые должны быть доступны в разных частях приложения, хранятся в едином глобальном хранилище — `store`. Это позволяет компонентам получать состояние напрямую из хранилища и реагировать на его изменения через подписку.

_Преимущества_: однозначность источника правды, упрощённое взаимодействие между компонентами, облегчённая отладка.

##### Immutable state updates

`Redux` поощряет неизменяемость состояния. Каждое обновление состояния создаёт новый объект, что предотвращает непреднамеренные мутации и упрощает отслеживание изменений. `Redux Toolkit` автоматически использует библиотеку `Immer`, позволяя писать обновления «как будто мутирующие», но на деле сохраняя неизменяемость.

_Преимущества_: безопасные изменения состояния, лёгкость интеграции с DevTools, предсказуемое поведение приложения.

##### Синхронизация с localStorage через `redux-persist`

Для сохранения состояния между перезагрузками страницы используется `redux-persist`, который автоматически сохраняет нужные части `store` в `localStorage` и восстанавливает их при инициализации приложения.

_Преимущества_: улучшенный пользовательский опыт, минимизация потери данных при перезагрузке, простота восстановления состояния.

#### **Контроллеры (api/controllers/)**

##### Auth Controller (`auth.controller.js`)

Отвечает за аутентификацию пользователей:

| Функция   | Метод | Endpoint            | Описание                          |
| --------- | ----- | ------------------- | --------------------------------- |
| `signup`  | POST  | `/api/auth/signup`  | Регистрация нового пользователя   |
| `signin`  | POST  | `/api/auth/signin`  | Вход в систему (email/password)   |
| `google`  | POST  | `/api/auth/google`  | OAuth аутентификация через Google |
| `signout` | GET   | `/api/auth/signout` | Выход из системы                  |

**Особенности:**

- Пароли хешируются с помощью bcryptjs (salt rounds: 10)
- JWT токен сохраняется в HTTP-only cookie
- Google OAuth создает пользователя автоматически, если не существует
- При Google OAuth генерируется случайный пароль

##### User Controller (`user.controller.js`)

Управление пользователями и их данными:

| Функция          | Описание                                                           |
| ---------------- | ------------------------------------------------------------------ |
| `updateUser`     | Обновление данных пользователя (username, email, password, avatar) |
| `deleteteUser`   | Удаление аккаунта и всех связанных объявлений                      |
| `getUserListing` | Получение всех объявлений конкретного пользователя                 |
| `getUser`        | Получение информации о пользователе по ID                          |

**Безопасность:**

- Проверка, что пользователь может изменять только свои данные
- При удалении пользователя каскадно удаляются все его объявления
- Пароли не возвращаются в ответах API

##### Listing Controller (`listing.controller.js`)

Управление объявлениями о недвижимости:

| Функция         | Описание                                               |
| --------------- | ------------------------------------------------------ |
| `createListing` | Создание нового объявления                             |
| `updateListing` | Обновление существующего объявления                    |
| `deleteListing` | Удаление объявления                                    |
| `getListing`    | Получение одного объявления по ID                      |
| `getListings`   | Получение списка объявлений с фильтрацией и пагинацией |

```
PUT    /api/listing/update/:id       // Обновление объявления (защищен)
GET    /api/listing/get/:id          // Получение объявления
GET    /api/listing/get              // Получение списка с фильтрами
```

#### **Утилиты (api/utils/)**

##### Error Handler (`error.js`)

Функция длpя создания стандартизированных ошибок:

##### Verify Token Middleware (`verifyUser.js`)

Middleware для проверки JWT токена:

- Извлекает токен из cookie `access_token`
- Проверяет валидность токена
- Добавляет данные пользователя в `req.user`
- Возвращает 401 (Unauthorized) если токен отсутствует
- Возвращает 403 (Forbidden) если токен невалиден

### **Безопасность**

- **HTTP-only cookies** — защита от XSS атак (JavaScript не может получить токен)
- **Bcrypt hashing** — необратимое хеширование паролей
- **JWT Secret** — хранится в переменных окружения
- **Authorization checks** — пользователь может изменять только свои данные
- **Error handling** — стандартизированная обработка ошибок
- **Mongoose validation** — валидация на уровне схемы

## Frontend (Client)

### Структура Frontend

#### **Точка входа (client/src/main.jsx)**

Инициализирует React приложение:

- Оборачивает приложение в Redux Provider
- Подключает PersistGate для сохранения состояния
- Монтирует приложение в DOM

#### **Маршрутизация (client/src/App.jsx)**

Определяет все маршруты приложения:

| Путь                         | Компонент     | Защищен | Описание                  |
| ---------------------------- | ------------- | ------- | ------------------------- |
| `/`                          | Home          | Нет     | Главная страница          |
| `/sign-up`                   | SignUp        | Нет     | Страница регистрации      |
| `/sign-in`                   | SignIn        | Нет     | Страница входа            |
| `/about`                     | About         | Нет     | О проекте                 |
| `/listing/:listingId`        | Listing       | Нет     | Просмотр объявления       |
| `/search`                    | Search        | Нет     | Поиск объявлений          |
| `/profile`                   | Profile       | Да      | Профиль пользователя      |
| `/create-listing`            | CreateListing | Да      | Создание объявления       |
| `/update-listing/:listingId` | UpdateListing | Да      | Редактирование объявления |

**Защищенные маршруты** используют компонент `PrivateRoute`, который проверяет наличие авторизованного пользователя в Redux store.

#### **Redux State Management**

##### Store Configuration (`redux/store.js`)

Конфигурация Redux store:

- Использует `redux-persist` для сохранения в localStorage
- Отключает проверку сериализуемости для корректной работы с persist
- Ключ хранения: `root`
- Версия: 1

##### User Slice (`redux/user/userSlice.js`)

Отвечает за управление состоянием пользователя в приложении. Он обеспечивает централизованное хранение данных о текущем пользователе, обработку ошибок и контроль состояния загрузки:

**Состояние:**

| Поле          | Тип            | Описание                                                                                               |
| ------------- | -------------- | ------------------------------------------------------------------------------------------------------ |
| `currentUser` | Object \| null | Данные текущего пользователя; `null`, если пользователь не авторизован                                 |
| `error`       | Object \| null | Информация об ошибках при загрузке или обновлении данных пользователя                                  |
| `loading`     | Boolean        | Индикатор состояния загрузки данных пользователя; `true` — идет загрузка, `false` — загрузка завершена |

## Аутентификация

### Регистрация:

1. Пользователь отправляет данные на `/api/auth/signup`
2. Пароль хешируется с помощью bcryptjs (10 salt rounds)
3. Создается новая запись в базе данных
4. Возвращается сообщение об успехе

### Вход:

1. Пользователь отправляет email и password на `/api/auth/signin`
2. Проверяется существование пользователя
3. Сравнивается хеш пароля с помощью bcryptjs
4. Генерируется JWT токен с payload `{ id: user._id }`
5. Токен сохраняется в HTTP-only cookie с именем `access_token`
6. Возвращаются данные пользователя (без пароля)

### Google OAuth:

1. Пользователь проходит аутентификацию через Firebase (фронтенд)
2. Данные отправляются на `/api/auth/google`
3. Проверяется существование пользователя по email
4. Если пользователь существует — выполняется вход
5. Если нет — создается новый пользователь с:
   - Сгенерированным username (на основе имени + случайные символы)
   - Случайным паролем
   - Avatar из Google аккаунта
6. Генерируется JWT и сохраняется в cookie

### Защищенные запросы:

1. Middleware `verifyToken` извлекает токен из cookie
2. Проверяет валидность и расшифровывает payload
3. Добавляет данные пользователя в `req.user`
4. Передает управление следующему middleware/контроллеру

### Выход:

1. Запрос на `/api/auth/signout`
2. Cookie `access_token` очищается
3. Фронтенд очищает Redux state
